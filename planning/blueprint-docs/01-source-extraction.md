# Source Extraction

## 1. このシステムは何か

既存ドキュメントから抽出できる Grift の定義は、単なる見積ツールではない。

Grift は、受託開発会社や小規模 AI 開発チーム向けの「上流工程インテリジェンス基盤」である。  
`README.md` では、GitHub の実績データと市場エビデンスを使って AI 開発の適正価格をリアルタイム算出するプロダクトとして説明されている。  
`docs/v2/architecture-overview.md` では、初期ヒアリング、追加開発やバグ報告の整理、要件定義書、見積、提案、承認、Linear handoff までを一気通貫で扱う業務基盤として再定義されている。

LP では「AI で見積を自動化するサービス」よりも、次のように言い換える方が本質に近い。

- AI 開発の価格を、勘や営業トークではなく実績と市場証拠で決めるシステム
- ヒアリング、仕様化、見積、承認、handoff を一本化する上流工程 OS
- 受託開発会社が勝てる案件と勝てない案件を、データで見極める intelligence product

## 2. 誰向けのシステムか

一次のターゲットは明確に `README.md` に書かれている。

### Primary

- ソロ開発者
- 1-10 人規模の AI 開発チーム
- フリーランスエンジニア
- 小規模スタートアップ

彼らの課題:

- AI 開発の価格が案件ごとに揺れる
- 自分たちの技術力を言語化しにくい
- 営業資料に使える客観データがない
- 値付けで弱気になりやすい

### Secondary

- 発注側企業の PM
- CTO

彼らの課題:

- ベンダー見積が妥当か分からない
- 複数提案の比較軸がない
- 技術ベンダーの実力を定量的に判断しづらい

### Future

- 東南アジア SIer
- オフショア企業

ここでは「技術力の客観的誇示」と「先進国クライアントとの適正取引」が主な価値として示されている。

### 市場の前提

`docs/v2/README.md` と `docs/v2/adr-0006-deployment-topology.md` では、当面の運用対象を日本市場に限定すると明記されている。  
つまり、初期 LP は日本語ファースト、日本の受託商流、日本の価格交渉、日本の上流工程の不透明さを前提に設計するべきである。

## 3. 何を解決するのか

既存ドキュメントを横断すると、Grift が解こうとしている問題は 4 層ある。

### 3.1 値付けの不透明さ

`README.md` の冒頭メッセージが最も強い。

- AI development pricing is a grift
- 情報の非対称性が価格を歪めている
- GitHub 実績と市場エビデンスで適正価格を出す

LP での主要テーマはここになる。  
単なる「見積作成の効率化」ではなく、「価格の不透明さを壊す」が最も強いフック。

### 3.2 上流工程の属人化

`docs/v2/architecture-overview.md` では、Grift v2 は受託開発会社における上流工程の不確実性を下げる業務基盤とされている。

対象業務:

- 新規顧客の初期ヒアリング
- 追加開発 / 改修 / バグ報告の整理
- 要件定義書、見積、提案、承認、handoff

つまり、LP では「見積もりツール」だけを前面に出すと狭すぎる。  
正しくは「受注前後の意思決定を整えるシステム」。

### 3.3 技術力の可視化不足

`README.md` と `docs/v2/architecture-overview.md` では、GitHub リポジトリ発見、Velocity Metric、Repository Intelligence が重要コンテキストとして扱われている。

ここから読み取れる価値:

- GitHub 実績を営業の言葉ではなく数値で提示できる
- 類似案件との比較素材を生成できる
- 開発速度や再現性を主観ではなく履歴で示せる

これは小規模チームにかなり刺さる。  
「大手ほど肩書きや事例が強くないチームでも、GitHub 実績で戦える」という構図を作れるため。

### 3.4 学習する見積と継続的な精度改善

`docs/v2/adr-0004-operational-intelligence-loop.md` では、見積後の実績データを BigQuery に蓄積し、次回見積へ補正を返す closed loop が差別化要素として定義されている。

価値:

- 案件を重ねるほど見積精度が上がる
- 顧客ごとの癖や工数のぶれを学習できる
- 市場エビデンスの品質評価までループに乗る

LP では「一回きりの AI 見積」ではなく、「使うほど精度が上がる見積インテリジェンス」として見せるべき。

## 4. 主機能の抽出

ルート README と v2 ドキュメントから、LP に載せるべき主機能は以下に整理できる。

| 機能 | 抽出根拠 | LP での見せ方 |
|---|---|---|
| AI ヒアリング | `README.md`, `architecture-overview.md` | 曖昧な相談を要件に変える |
| 自動仕様書生成 | `README.md` | 会話から仕様書の叩き台を生成 |
| リアルタイム見積 | `README.md` | 工数、市場相場、自社実績を統合 |
| GitHub 実績分析 | `README.md`, `architecture-overview.md` | 技術力を証明する営業資産 |
| マルチソース市場調査 | `adr-0002` | 複数ソースで裏づけた相場証拠 |
| Three-Way Proposal | `architecture-overview.md`, `adr-0004`, `adr-0012` | 自社実績、市場相場、提案価格を同画面比較 |
| Go / No-Go 判定 | `README.md`, `adr-0004` | 受けるべき案件かを判断 |
| Operational Intelligence | `adr-0004` | 見積と実績の差から精度を改善 |
| Cross-Tenant Benchmark | `adr-0012`, `cross-tenant-intelligence-architecture.md` | 匿名ベンチマークで相場観を持てる |
| Linear / GitHub handoff | `README.md`, `architecture-overview.md` | 合意した内容をそのまま実行フェーズへ渡せる |

## 5. 差別化要因の抽出

### 5.1 実績データと市場データの二面性

多くの見積ツールは、工数テンプレートか LLM 要約に寄る。  
Grift は `README.md` と `adr-0002` により、以下の二面性を持つ。

- 内部根拠: GitHub 実績、Velocity、過去案件、実績補正
- 外部根拠: 複数プロバイダの市場調査、引用付き evidence

この 2 つを同時に持つことが、強い訴求点になる。

### 5.2 見積だけで終わらない

`architecture-overview.md` と `README.md` では、承認、handoff、Linear / GitHub 連携まで含まれている。  
したがって、LP は「見積作成 SaaS」に閉じない方がよい。

正しい認識:

- 受注前の認識合わせ
- 受注可否の判断
- 受注後の handoff

までを管理する。

### 5.3 継続学習する intelligence product

`adr-0004` と `adr-0012` が示すのは、Grift が workflow SaaS ではなく intelligence product を目指している点である。

- tenant ごとの精度改善ループ
- 匿名横断 benchmark
- pricing index
- demand radar
- stack trend

LP ではこの将来像を入れてよいが、出し方は慎重にする必要がある。  
現段階で promise しすぎず、「設計思想として持っている intelligence layer」として見せるのが適切。

## 6. LP に使うべきメッセージの芯

既存ドキュメントから最も一貫しているメッセージは次の 5 つ。

1. AI 開発の価格決定はブラックボックス化している
2. Grift は GitHub 実績と市場証拠でそのブラックボックスを壊す
3. 曖昧な相談を、仕様書・見積・提案に変換できる
4. 受けるべき案件かどうかまで判断できる
5. 使うほど見積精度が上がる

## 7. LP に入れない方がよい要素

設計資料には魅力的だが、初回 LP で前面に出しすぎると理解を阻害する内容もある。

- `Qwen3.5`, `vLLM`, `Pub/Sub`, `GKE` などの実装詳細
- Bounded Context の細分化
- tenant analytics の細かいデータモデル
- 学習データ lake の内部設計

これらは信頼補強として使うならよいが、ファーストビューや主導線には重い。  
LP の主軸はあくまで「価格の透明化」と「上流工程の意思決定支援」に置くべき。

## 8. Source-to-Landing 変換ルール

抽出内容を LP に落とす際の変換ルールを固定する。

| 元情報 | LP 上の翻訳 |
|---|---|
| 上流工程の不確実性を下げる | 相談から見積までの迷いを減らす |
| GitHub 実績分析 | 技術力を営業資料に変える |
| Market Benchmark | 相場を複数ソースで裏づける |
| Three-Way Proposal | 相場、自社実績、提案額を同時比較 |
| Operational Intelligence | 使うほど見積精度が上がる |
| Cross-Tenant Intelligence | 匿名ベンチマークで市況を掴める |

この翻訳を守ると、ドキュメントの中身を損なわず、非技術ユーザーにも通じる表現になる。
