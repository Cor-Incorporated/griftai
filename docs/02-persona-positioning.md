# Persona And Positioning

## 1. ポジショニングの基本方針

Grift の LP は、広く「AI を使った開発支援」と見せるより、次の一点に絞った方が刺さる。

> AI 開発の見積と提案を、勘ではなく実績と市場証拠で作るための intelligence platform

この言い方であれば、README の強い主張と v2 ドキュメントの業務基盤像の両方を回収できる。

## 2. 優先ペルソナ

### Persona A: 小規模 AI 受託チームの代表

最重要ペルソナ。  
1-10 人規模、または個人から小規模法人に成長した開発チームの意思決定者。

#### 状況

- AI 系の案件相談は増えている
- ただし、毎回見積りの作り方がぶれる
- 値付けで強く出る根拠がない
- 大手より実績の見せ方で負けやすい

#### 本音

- 安く取りすぎたくない
- でも高く出して失注するのも怖い
- 「うちが妥当な価格です」と数字で言いたい
- GitHub 上の実績を営業武器にしたい

#### LP で刺さる論点

- 値付けの自信を持てる
- GitHub 実績を営業資料に変えられる
- 曖昧な相談を仕様書と見積に変えられる
- 受けるべき案件かを判断できる
- 使うほど見積精度が上がる

#### 欲しい証拠

- どういう根拠で価格を出すのか
- 市場相場をどう裏づけるのか
- GitHub 分析で何が見えるのか
- 見積後にどう精度改善されるのか

#### 刺さる CTA

- デモを見る
- 自社の GitHub 実績でどこまで見えるか確認する
- ベータ参加を申し込む

### Persona B: 発注側の PM / CTO

第二優先。  
ベンダー選定側、または見積査定の責任を負う立場。

#### 状況

- 複数ベンダーから提案が来る
- 価格差の理由が説明されない
- 開発力の判断が肩書き頼みになりやすい

#### 本音

- 過大見積もりを避けたい
- 妥当な相場感を持ちたい
- ベンダーの実力を客観的に比較したい

#### LP で刺さる論点

- 第三者的に相場をチェックできる
- 複数ソースの market evidence がある
- GitHub 実績と Velocity を見て比較できる
- 見積理由がブラックボックスではない

#### 欲しい証拠

- Three-Way Proposal の例
- benchmark の例
- 引用付き evidence のイメージ

### Persona C: 将来の海外 SIer / オフショア企業

将来向け。初期 LP の主役にはしないが、拡張性として残す。

#### 刺さる論点

- 技術力の客観証明
- 海外クライアントに対する価格の透明性
- 匿名 benchmark による交渉材料

## 3. ペルソナ別 JTBD

### Persona A の JTBD

- 案件相談が来たとき、安売りせずに納得感ある見積を返したい
- 自社の強みを感覚ではなくデータで説明したい
- 相談から仕様書、見積、handoff までの前工程を標準化したい

### Persona B の JTBD

- ベンダー見積が妥当か短時間で判断したい
- 複数提案の違いを根拠つきで比較したい
- 実行力のあるチームかどうかを事前に見抜きたい

## 4. ポジショニング文

### 短い版

Grift は、AI 開発の見積と提案を実績データと市場証拠で作るための platform。

### 長い版

Grift は、小規模 AI 開発チームや受託開発会社が、曖昧な案件相談を仕様書、見積、提案に変え、GitHub 実績と市場エビデンスに基づいて適正価格を提示するための上流工程インテリジェンス基盤である。

## 5. LP のメッセージ優先順位

### 1位: 価格の透明化

最初に伝えるべきこと。  
「AI 開発の価格はぶれすぎる。そのぶれを壊す」が主フック。

### 2位: 実績で戦える

小規模チームが最も反応しやすい。  
「営業が強い会社」ではなく「実績がある会社」が勝ちやすくなる、という構図を作る。

### 3位: 相談から仕様化までつながる

単なる価格計算機ではなく、ヒアリングから見積書生成までつながることを示す。

### 4位: 学習する精度

Operational Intelligence を簡潔に翻訳し、「使うほど精度が上がる」と見せる。

### 5位: 匿名 benchmark の将来価値

これは信頼補強として中盤以降に出す。  
ヒーローで大きく出すと概念が重くなる。

## 6. 刺さる言葉

- 適正価格
- 実績ベース
- 相場の根拠
- GitHub で証明する
- 曖昧な相談を要件に変える
- 提案に耐える見積
- 受けるべき案件が分かる
- 使うほど精度が上がる

## 7. 避ける言葉

- 万能 AI エージェント
- 完全自動化
- すべての開発会社向け
- LLM オーケストレーション
- マルチエージェント基盤
- データレイク

これらは技術的には間違っていなくても、LP としては焦点をぼかす。

## 8. 主要な反論と返し方

### 反論 1

「結局 AI の見積なんて信用できないのでは」

返し方:

- 市場相場だけでなく GitHub 実績を根拠に使う
- 複数ソースを cross-validate する
- 実績との差分を学習して精度を上げる

### 反論 2

「小規模チームにこんな仕組みは重いのでは」

返し方:

- 小規模だからこそ、営業資料と見積精度の標準化が効く
- 人を増やさず、上流工程の再現性を上げる

### 反論 3

「機密情報や顧客データは大丈夫か」

返し方:

- tenant 分離
- raw text を含まない匿名 benchmark
- opt-in 前提の cross-tenant intelligence

## 9. LP の勝ち筋

初回 LP で狙うべき勝ち筋は、SaaS 一般論ではない。

- 受託の値付けに自信がない人の不安に直撃する
- 実績を営業武器に変えたい人に具体像を与える
- 見積の根拠を求める発注側にも一定の信頼を作る

この 3 点が同時に成立すると、Grift は単なる業務効率化ツールではなく「交渉力を持つためのシステム」として立ち上がる。
