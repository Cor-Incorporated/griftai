# Content Strategy

## 1. コンテンツ戦略の前提

この LP で一番やってはいけないのは、機能紹介の羅列にすること。  
Grift は機能数よりも、価格の透明化という論点で記憶されるべきである。

したがって、コピーは次の順序で組む。

1. 問題を切る
2. 解決の仕組みを示す
3. 根拠を見せる
4. 信頼不安を潰す
5. 行動を促す

## 2. Hero で書くべきこと

### 主張

- AI 開発の価格はブラックボックス化している
- Grift はそれを壊す

### 良いヘッドラインの条件

- 価格の歪みを直接言う
- 受託の現場にある怒りや不信を代弁する
- 何を解決するかが 1 文で分かる

### ヘッドライン案

- AI開発の見積を、勘ではなく根拠で出す。
- AI受託の価格決定を、GitHub実績と市場証拠で透明化する。
- その見積、本当に妥当ですか。Grift は根拠を返します。

### サブコピー案

- 曖昧な案件相談を、仕様書・見積・提案に変える。
- 自社の GitHub 実績と複数ソースの市場 evidence を統合し、適正価格を提示する。
- 小規模 AI チームの提案力を、感覚ではなくデータで底上げする。

### CTA 案

- ベータに参加する
- デモを見る
- 自社に合うか確認する

## 3. Problem セクションで書くべきこと

このセクションは、見積作業そのものではなく「価格交渉の不公平」を描く。

### 書くべき論点

- AI 案件は要件が曖昧なまま相談が始まる
- 見積額が営業の強さや経験値でぶれやすい
- 発注側も受託側も妥当性を確認しづらい
- 小規模チームほど実績の見せ方で不利になる

### 見せ方

- 3 つの痛みを短く並べる
- 数字ではなく状況描写で共感を取る

### 例文

- 要件が曖昧なまま見積を迫られる
- 妥当な価格なのに、根拠を示せず値切られる
- 実力はあるのに、営業資料が弱くて負ける

## 4. Solution セクションで書くべきこと

ここでは「AI が全部やります」と言わない。  
Grift が根拠を集め、構造化し、提案に耐える形へ変換することを示す。

### 書くべき流れ

1. 相談内容をヒアリングする
2. 要件を仕様に整理する
3. GitHub 実績を分析する
4. 市場 evidence を集める
5. 提案できる見積にする

### 例文

- 会話から、案件の曖昧さを減らす
- リポジトリ履歴から、開発力を営業可能な形に変える
- 複数ソースの相場証拠で、価格に説明責任を持たせる

## 5. Three-Way Proposal で書くべきこと

このセクションは LP のコア。  
ここが弱いと Grift 固有の価値が見えない。

### 必ず含める 3 軸

- 自社実績
- 市場相場
- 最終提案

### 伝えるべき意味

- 相場だけではない
- 自社都合だけでもない
- 両方を見た上で提案額が決まる

### コピー例

- 相場に寄せるだけではなく、自社の実行力も反映する
- 価格の根拠を、ひとつの画面で比較できる
- 提案額が「なんとなく」に見えない

## 6. GitHub Intelligence で書くべきこと

このセクションは、Primary persona に最も効く。

### 強調すべきこと

- GitHub 実績を営業資料に変える
- 開発速度や継続性を可視化する
- 類似案件に対する再現性を示す

### 刺さるコピー例

- 実績を、口頭説明ではなく履歴で証明する
- 開発力を「速い気がする」から「数字で示せる」に変える
- 小さなチームでも、GitHub の積み上げを提案力に変えられる

## 7. Market Evidence で書くべきこと

このセクションは信頼補強。  
価格の正しさを「市場に照らして」説明できることを見せる。

### 書くべきこと

- 1 ソース依存ではない
- 複数の独立ソースを cross-validate する
- 根拠に citation がある
- 矛盾時には人間レビュー前提で扱う

### 刺さるコピー例

- 相場感ではなく、相場の根拠を残す
- 複数ソースで裏づけた価格レンジを参照する
- 証拠のない強気価格も、過度な安売りも避けやすくなる

## 8. Intelligence Loop で書くべきこと

Operational Intelligence は、LP では難しい言葉を捨ててよい。

### 言い換え

- 見積と実績の差を学習する
- 案件を重ねるほど補正が効く
- 次の提案がより現実に近づく

### コピー例

- 出した見積で終わらない。実績との差を次に返す。
- 使うほど、あなたのチームに合った見積精度へ寄っていく。

## 9. Trust セクションで書くべきこと

信頼は後付けではなく、LP 中盤で明示した方がよい。  
特に GitHub や案件情報を扱う以上、データ不安は先回りして潰す。

### 書くべき論点

- tenant 分離
- 匿名 benchmark
- raw text を横断利用しない
- opt-in 前提の cross-tenant intelligence

### コピー例

- 顧客データをそのまま横断利用しない
- benchmark は匿名化と opt-in を前提に設計
- 価値化とガバナンスを分けて考える

## 10. FAQ で回収すべき質問

- AI の見積はどこまで信用できますか
- GitHub を連携しないと使えませんか
- 顧客データは学習に使われますか
- 発注側でも使えますか
- どの規模のチーム向けですか

## 11. セクション順の推奨

1. Hero
2. Problem
3. Solution
4. Three-Way Proposal
5. GitHub Intelligence
6. Market Evidence
7. Workflow
8. Intelligence Loop
9. Trust
10. FAQ
11. CTA

この順番にすると、問題認識から信頼形成までが素直につながる。

## 12. クリエイティブの方向性

### 向いている表現

- タイポグラフィ中心
- 図解と比較表
- 数字、帯、レンジ表現
- 落ち着いた配色
- 少数の強いメッセージ

### 向いていない表現

- AI っぽい抽象グラデーション
- 汎用 SaaS のヒーローイラスト
- ネオン、glow、過度な近未来感
- 技術スタックのロゴ羅列

Grift の信頼は、派手さではなく「根拠を扱うプロダクトらしさ」から出した方が強い。

## 13. 最低限必要なアセット

- Three-Way Proposal のモック
- GitHub Intelligence のモック
- Workflow の図
- benchmark 表現のモック
- CTA 用の簡易フォーム

## 14. 最終的にユーザーに残したい印象

- これは AI のお遊びではなく、提案に耐える仕組みだ
- 小規模チームでも、実績で戦えるようになりそうだ
- 価格の説明責任を持てそうだ
- 一度使ってみたい
